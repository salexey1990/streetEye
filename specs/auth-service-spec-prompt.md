# Prompt: Написание спецификации Auth Service для StreetEye

## Контекст и роль

Ты — **Senior Backend Architect** с 10+ годами опыта проектирования распределённых систем и микросервисной архитектуры. Твоя специализация — системы аутентификации и авторизации для высоконагруженных мобильных приложений.

Ты работаешь над проектом **StreetEye** — мобильным приложением-тренажёром для стрит-фотографов.

## Задача

Напиши **полную, детальную спецификацию Auth Service** — микросервиса аутентификации и авторизации для платформы StreetEye.

## Входные данные

### 1. Общая информация о проекте

Прочитай и проанализируй следующие документы:

- `specs/spec.md` — общая спецификация проекта (цели, ЦА, функциональность)
- `specs/backend-spec.md` — архитектура бэкенда, описание всех микросервисов
- `specs/challenge-service-spec.md` — пример спецификации другого сервиса (используй как образец структуры)

### 2. Требования из существующих документов

Из `backend-spec.md` извлеки следующую информацию:

```
Auth Service (порт 3001):
- Зона ответственности: регистрация, аутентификация, JWT токены, refresh token ротация
- Основные endpoints: register, login, refresh, logout, password/reset
- Взаимодействие: User Service, Notification Service, Analytics Service
- Технологии: NestJS, JWT, bcrypt, PostgreSQL, Redis
```

### 3. Требования к безопасности

Учти следующие требования безопасности:

- **Пароли:** bcrypt с cost factor 12 или argon2id
- **Токены:** JWT (access token 15 мин, refresh token 7 дней)
- **Rate limiting:** защита от brute force (5 попыток регистрации/мин, 10 попыток входа/мин)
- **Email верификация:** обязательна для новых пользователей
- **2FA:** опционально для Premium+ пользователей (TOTP)
- **Блокировки:** автоматическая блокировка после 10 неудачных попыток входа
- **Хранение токенов:** refresh tokens в Redis с TTL, blacklist для отозванных токенов

**Обязательные endpoints:**

**Аутентификация:**
- `POST /auth/register` — регистрация нового пользователя
- `POST /auth/login` — вход
- `POST /auth/logout` — выход
- `POST /auth/refresh` — обновление access token
- `POST /auth/verify-email` — подтверждение email
- `POST /auth/resend-verification` — повторная отправка подтверждения

**Восстановление доступа:**
- `POST /auth/password/reset` — запрос на сброс пароля
- `POST /auth/password/reset/:token` — установка нового пароля

**2FA (опционально):**
- `POST /auth/2fa/enable` — включение 2FA
- `POST /auth/2fa/disable` — отключение 2FA
- `POST /auth/2fa/verify` — подтверждение 2FA кода

**Управление сессиями:**
- `GET /auth/sessions` — список активных сессий
- `DELETE /auth/sessions/:id` — завершение сессии
- `DELETE /auth/sessions/all` — завершение всех сессий

**Admin endpoints:**
- `POST /auth/users/:id/block` — блокировка пользователя
- `POST /auth/users/:id/unblock` — разблокировка

## Дополнительные указания

### Интеграция с другими сервисами

Учти следующие взаимодействия:

1. **User Service**
   - Создание пользователя при регистрации
   - Получение данных пользователя для токена
   - Обновление last_login_at

2. **Notification Service**
   - Отправка email верификации
   - Отправка password reset email
   - Уведомления о безопасности (новый вход, смена пароля)

3. **Analytics Service**
   - Событие `user.registered`
   - Событие `user.logged_in`
   - Событие `user.login_failed`
   - Событие `user.password_reset`

4. **Redis**
   - Хранение refresh токенов
   - Blacklist отозванных токенов
   - Rate limiting counters
   - Email verification codes

### Особые сценарии

Опиши подробно следующие сценарии:

1. **Регистрация с верификацией**
   - Пользователь не может войти без подтверждённого email
   - Код верификации действителен 24 часа
   - После 3 неудачных попыток — блокировка на 1 час

2. **Refresh token ротация**
   - При каждом использовании refresh token выдаётся новая пара
   - Старый refresh token добавляется в blacklist
   - Если используется старый токен — все сессии отзываются

3. **Восстановление доступа**
   - Токен сброса пароля действителен 1 час
   - После успешного сброса — все сессии отзываются
   - Отправка уведомления о смене пароля

4. **Блокировка при brute force**
   - 10 неудачных попыток — блокировка на 15 минут
   - Блокировка по IP + email
   - Уведомление пользователя о попытке взлома
