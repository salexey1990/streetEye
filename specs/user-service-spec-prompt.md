# Prompt: Написание спецификации User Service для StreetEye

## Контекст и роль

Ты — **Senior Backend Architect** с 10+ годами опыта проектирования распределённых систем и микросервисной архитектуры. Твоя специализация — системы управления пользователями, подписками и монетизации для мобильных приложений.

Ты работаешь над проектом **StreetEye** — мобильным приложением-тренажёром для стрит-фотографов.

## Задача

Напиши **полную, детальную спецификацию User Service** — микросервиса управления профилями пользователей, подписками и тарифами для платформы StreetEye.

## Входные данные

### 1. Общая информация о проекте

Прочитай и проанализируй следующие документы:

- `specs/spec.md` — общая спецификация проекта (цели, ЦА, функциональность, монетизация)
- `specs/backend-spec.md` — архитектура бэкенда, описание всех микросервисов
- `specs/challenge-service-spec.md` — пример спецификации другого сервиса (используй как образец структуры)
- `specs/auth-service-spec.md` — пример детальной спецификации Auth Service

### 2. Требования из существующих документов

Из `backend-spec.md` извлеки следующую информацию:

```
User Service (порт 3002):
- Зона ответственности: профили пользователей, подписки, тарифы (Free/Premium/Masterclass)
- Основные endpoints: profile (GET/PUT), subscription (GET/POST/PUT), purchases
- Взаимодействие: Auth Service, Notification Service, Analytics Service, Payment Gateway

```

Из `spec.md` извлеки информацию о монетизации:

### 3. Требования к безопасности и данным

Учти следующие требования:

- **Персональные данные:** GDPR/CCPA compliance, шифрование PII
- **Подписки:** Интеграция с платёжными провайдерами (Stripe, Apple IAP, Google Play Billing)
- **Кэширование:** Профили пользователей в Redis с TTL 5 минут
- **Аудит:** Логирование всех изменений подписок и покупок
- **Rate limiting:** 60 запросов профиля/мин, 10 запросов изменения/мин

Для **каждого endpoint** укажи:

```markdown

```
METHOD: GET|POST|PUT|DELETE|PATCH

  ...
}

  field: type,
  ...
}

- CODE: описание
- CODE: описание

**Обязательные endpoints:**

**Профиль пользователя:**
- `GET /users/:id/profile` — получение профиля
- `PUT /users/:id/profile` — обновление профиля
- `GET /users/:id/settings` — настройки пользователя
- `PUT /users/:id/settings` — обновление настроек
- `DELETE /users/:id/account` — удаление аккаунта (GDPR)

**Подписки:**
- `GET /users/:id/subscription` — текущая подписка
- `POST /users/:id/subscription/upgrade` — улучшение подписки
- `POST /users/:id/subscription/cancel` — отмена подписки
- `POST /users/:id/subscription/restore` — восстановление подписки

**Тарифы:**
- `GET /subscription/plans` — список доступных тарифов
- `GET /subscription/plans/:id` — детали тарифа

**Покупки (Masterclass):**
- `GET /users/:id/purchases` — история покупок
- `POST /users/:id/purchases/courses/:courseId` — покупка курса
- `GET /users/:id/purchases/courses` — доступные курсы

**Статистика:**
- `GET /users/:id/stats` — статистика пользователя
- `GET /users/:id/achievements` — достижения

**Admin endpoints:**
- `GET /admin/users` — список пользователей (admin)
- `GET /admin/users/:id` — детали пользователя (admin)
- `POST /admin/users/:id/ban` — блокировка (admin)
- `POST /admin/users/:id/unban` — разблокировка (admin)

## Стиль и формат

### Требования к стилю

1. **Профессиональный технический язык**
   - Избегай разговорных выражений
   - Используй терминологию предметной области
   - Пиши кратко и по делу

2. **Структурированность**
   - Используй заголовки H1-H4
   - Применяй списки и таблицы
   - Вставляй диаграммы Mermaid где уместно

3. **Примеры**
   - Для каждого endpoint приводи примеры request/response
   - Используй реалистичные данные (не "foo", "bar")
   - Показывай как успешные, так и ошибочные сценарии

4. **Валидация**
   - Для всех полей указывай ограничения (min/max, regex, enum)
   - Описывай все возможные ошибки
   - Указывай rate limits

### Объём

**Целевой объём:** 1500-2500 строк (как `challenge-service-spec.md`)

**Баланс разделов:**
- Архитектура: 15-20%
- API: 40-50%
- База данных: 20-25%
- Реализация: 15-20%
- Ошибки: 5-10%

## Дополнительные указания

### Интеграция с другими сервисами

Учти следующие взаимодействия:

1. **Auth Service**
   - Валидация JWT токенов
   - Получение user_id из токена
   - Синхронизация при регистрации

2. **Notification Service**
   - Уведомления об изменении подписки
   - Напоминания о продлении
   - Подтверждения покупок

3. **Analytics Service**
   - Событие `user.profile_updated`
   - Событие `subscription.upgraded`
   - Событие `purchase.completed`

4. **File Service**
   - Загрузка аватара пользователя
   - CDN URLs для изображений

5. **Payment Gateway (внешний)**
   - Stripe API для подписок
   - Apple In-App Purchase
   - Google Play Billing

### Особые сценарии

Опиши подробно следующие сценарии:

1. **Удаление аккаунта (GDPR)**
   - Экспорт всех данных пользователя
   - Полное удаление PII
   - Сохранение анонимизированной статистики

2. **Изменение подписки**
   - Proration при улучшении/ухудшении
   - Сохранение доступа до конца периода
   - Webhook обработка от платёжного провайдера

3. **Восстановление подписки**
   - Проверка receipt от Apple/Google
   - Восстановление purchase history
   - Активация подписки

4. **Покупка курса**
   - Проверка наличия курса
   - Обработка оплаты
   - Предоставление доступа
